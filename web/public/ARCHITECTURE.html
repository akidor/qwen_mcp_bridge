로컬 vLLM이 자연어로 도시계획 데이터를 다루는 3-레포 파이프라인입니다. 사용자가 브라우저 채팅 UI에서 질의하면 qwen_mcp_bridge가 vLLM과 urban_mcp(8개 stdio MCP 도메인) 사이를 중계하고, urban_mcp는 HTTP로 northside_threejs의 polygon 백엔드(:7469)와 등시선 백엔드를 호출해 필지·규제·생성·분석 데이터를 PostgreSQL 16 + PostGIS에서 끌어옵니다.

03

컴포넌트별 역할

관심사를 LLM 오케스트레이션 · 도구 노출 · 도메인 연산으로 분리

./qwen_mcp_bridge/ · 오케스트레이터

◆ qwen_mcp_bridge

vLLM과 urban_mcp 8개 stdio MCP 도메인을 잇는 OpenAI 호환 HTTP 브릿지

FastAPI 앱(server.py)이 /healthz, /v1/models, /v1/chat/completions를 노출하고, mcp_pool.py가 8개 도메인을 동시 spawn해 tool dispatch를 처리합니다. chat_loop.py의 tool_call 루프(max_tool_iterations, 기본 12)가 vLLM의 도구 호출과 결과 반환을 반복하며, 도메인 일부가 spawn에 실패해도 나머지는 계속 동작합니다(per-domain try/except). web/에 Vite+React 채팅 UI가 있고 dev에서 :7474로 프록시합니다.

UI → bridge bridge → vLLM bridge → urban_mcp
./urban_mcp/ · 도구 계층

◆ urban_mcp

polygon(:7469)·등시선 백엔드를 8개 도메인 stdio MCP 서버로 감싸는 어댑터

브릿지가 질의 처리 시 uv --directory <urban_mcp_root> run python -m urban_mcp_<domain> 명령으로 각 도메인 프로세스를 spawn합니다. shared/ 패키지가 HTTP 클라이언트 (get_polygon_client(), get_isochrone_client())와 공통 타입·오류 처리를 제공하고, locate 도메인의 주소·시설 검색에는 KAKAO_REST_API_KEY가 필요합니다.

bridge → mcp (spawn) mcp → polygon mcp → 등시선
./northside_threejs/ · 연산 · 데이터

◆ northside_threejs

필지 규제를 반영한 건물 매싱 생성 백엔드(:7469) + 시각화 프론트(Next.js :6974)

polygon_generator/ 패키지가 셋백·정북사선·도로감지·지형 알고리즘으로 2D/3D 매스를 산출하고, whatif/ 하위 18개 서브도메인이 규제 시뮬레이션을 담당합니다. urban_mcp가 HTTP로 이 백엔드를 호출하며, 외부 인터페이스로 PostgreSQL 16 + PostGIS와 GeoServer WMS, MARTIN 타일 서버에 의존합니다.

mcp → polygon polygon → PostGIS WMS · MARTIN

06

전체 기동 순서

의존 백엔드 → MCP → 브릿지 → 헬스체크 → UI

vLLM(:8016), polygon 백엔드(:7469) 기동

urban_mcp 의존성 설치

cd urban_mcp
uv sync --all-packages

루트 pyproject.toml의 dependencies=[] 때문에 uv sync만으로는 멤버 패키지가 설치되지 않습니다. 반드시 --all-packages 필요.

브릿지 기동

cd qwen_mcp_bridge
uv sync --extra dev
uv run uvicorn qwen_mcp_bridge.server:app --host 0.0.0.0 --port 8090

헬스체크

curl localhost:8090/healthz

응답의 ready_domains가 8개, failed_domains가 {}인지 확인.

채팅 UI 기동

cd qwen_mcp_bridge/web
npm ci && npm run dev   # → http://localhost:7474
